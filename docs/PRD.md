## 一、产品定位

### 1.1 核心问题

| 痛点 | 现状 |
|------|------|
| C盘空间不足 | 用户不知道什么占空间 |
| 清理风险 | 怕删错东西，不敢动手 |
| 迁移复杂 | 符号链接、Junction 概念晦涩 |
| 反复出现 | 清理后不久又满了 |

### 1.2 目标用户

| 用户类型 | 特征 | 核心诉求 |
|---------|------|---------|
| 普通用户 | 不懂技术，C盘红了就慌 | 一键清理，安全可靠 |
| 开发者 | 装各种环境，目录复杂 | 精准迁移，保留数据 |
| 运维人员 | 管理多台机器 | 批量操作，可脚本化 |

### 1.3 产品形态

```
┌─────────────────────────────────────────────┐
│                Diskman                       │
├─────────────────────────────────────────────┤
│  CLI        │ 命令行工具，开发者/运维使用     │
│  MCP Server │ AI 集成，让 AI 帮用户操作       │
│  Python API │ 程序集成                        │
└─────────────────────────────────────────────┘
```

## 三、AI 模块设计

### 3.1 为什么需要 AI？

| 场景 | 规则引擎 | AI |
|------|---------|-----|
| "npm-cache 可以删" | ✅ 能判断 | ✅ 能判断 |
| "AppData\Local\某奇怪的目录" | ❌ 规则覆盖不到 | ✅ 根据目录名+文件类型推断 |
| "用户说我是做 Python 开发的" | ❌ 无法理解上下文 | ✅ 保留 Python 相关目录 |
| "这个目录看起来像项目代码" | ❌ 难以判断 | ✅ 分析文件结构推断 |
| 向用户解释为什么 | ❌ 只能输出固定文案 | ✅ 自然语言解释 |

### 3.2 AI 架构（简化版）

```
┌─────────────────────────────────────────────────────────┐
│                    MCP Server / Python API              │
│                                                         │
│   analyze_directories() ──→ AIService ──→ AI Provider   │
│         │                         │                     │
│         │ 自动切换                 ├─→ OpenAI           │
│         │                         ├─→ DeepSeek          │
│         ▼                         ├─→ Qwen              │
│   无 AI 配置 → 规则引擎           └─→ Ollama (本地)      │
│                                                         │
│   配置方式：                                             │
│   1. 参数传入（推荐）: AIService(AIConfig(...))         │
│   2. 环境变量: AI_API_KEY / AI_BASE_URL / AI_MODEL     │
└─────────────────────────────────────────────────────────┘
```

**关键设计决策：** 
- ~~不需要单独的 API Server~~
- MCP Server 直接调用 AI Service
- 配置支持参数传入（推荐）或环境变量
- 无 AI 配置时自动使用规则引擎

### 3.3 支持的 AI Provider

| Provider | 优点 | 缺点 | 配置 |
|----------|------|------|------|
| OpenAI | 效果好，API 稳定 | 贵，国内访问需代理 | `api_key="sk-xxx"` |
| DeepSeek | 便宜，国内可用 | 效果略差 | `base_url="https://api.deepseek.com"` |
| Qwen | 国内可用，便宜 | - | `base_url="https://dashscope.aliyuncs.com/compatible-mode/v1"` |
| Ollama | 无需联网，隐私 | 效果一般，需硬件 | `base_url="http://localhost:11434/v1"` |

### 3.4 智能分析模式

`analyze_directories` 工具会自动选择分析模式：

| 条件 | 分析模式 |
|------|----------|
| AI 配置传入 + 可用 + `prefer_ai=True` | AI 分析 |
| 无 AI 配置 / AI 不可用 / `prefer_ai=False` | 规则分析 |
| AI 分析失败 | 自动降级到规则分析 |


### 4.2 模块拆分

```
diskman/
├── operations/       # 文件系统操作
│   ├── scanner.py   # 扫描引擎（支持符号链接正确处理）
│   ├── migrator.py  # 迁移引擎
│   └── cleaner.py   # 清理引擎
│
├── analysis/         # 分析模块
│   ├── analyzer.py  # 规则引擎分析
│   └── rules/       # 内置规则
│
├── ai/              # AI 模块
│   ├── service.py   # AI 服务封装
│   └── providers/   # OpenAI 兼容的 Provider
│
├── mcp/             # MCP Server
│   └── server.py    # AI Agent 集成入口
│
└── cli.py           # 命令行界面
```

### 4.3 关键设计

| 设计点 | 方案 |
|--------|------|
| AI 集成 | 直接在 MCP/Python 中调用，无需单独 API Server |
| AI 配置 | 支持参数传入（推荐）或环境变量 |
| 符号链接处理 | 默认不统计符号链接目标大小，避免重复计算 |
| 执行安全 | 操作前确认 + dry run 默认开启 |
| 隐私保护 | 可选本地模型，数据不出本机 |
| 跨平台 | Windows 为主，未来支持 macOS |

## 五、配置方式

### 5.1 参数传入（推荐）

```python
from diskman import AIService, AIConfig

# 方式1：直接传入配置
ai = AIService(AIConfig(
    api_key="your-api-key",
    base_url="https://api.deepseek.com",
    model="deepseek-chat",
))

# 方式2：MCP Server 配置
from diskman.mcp import create_mcp_server

mcp = create_mcp_server(AIConfig(
    api_key="your-api-key",
    base_url="https://api.deepseek.com",
    model="deepseek-chat",
))
```

### 5.2 环境变量

| 变量 | 说明 | 示例 |
|------|------|------|
| `AI_API_KEY` | AI API 密钥 | `sk-xxx` |
| `AI_BASE_URL` | API 端点 | `https://api.deepseek.com` |
| `AI_MODEL` | 模型名称 | `deepseek-chat` |

或使用 OpenAI 兼容变量：`OPENAI_API_KEY`, `OPENAI_BASE_URL`, `OPENAI_MODEL`
